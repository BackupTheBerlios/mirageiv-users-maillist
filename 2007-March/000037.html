On 2/28/07, Scott Horowitz <stonecrest at gmail.com> wrote:
> 1. File menu item to open remote files
> 2. Some way to handle the file in the recent files list. Since /tmp
> will be cleared on a reboot, maybe this file should simply be skipped
> from the list.
> 3. Saving.. if the user chooses save, I think they should get the Save
> As dialog instead, as if there is no filename yet (the user doesn't
> care that the file is in /tmp).
> (Items 2 and 3 need to be implemented with the possibility that users
> might want to view images from /tmp, so you can't simply rely on the
> path starting with "/tmp" to dictate how things are handled.)

I made some progress on this tonight. I implemented items 1 and 2 -
Mirage now saves the file as /tmp/filename instead of using
/tmp/mirage-<hash> as in your patch, and there is some logic in there
to keep the files organized (e.g. if you open the same remote image
twice). The recent file list also retains the http/ftp link so that
the user will be able to bring up the recent file in the future (i.e.
after a reboot).

And because I changed the filename, I decided item 3 is no longer
needed. I think it's fine to get "tmp/logo.gif" whereas
"tmp/mirage-3ds8f787fsdf87sd8f.gif" wouldn't have been so nice.

The only remaining issue here is that the remote image should probably
be downloaded in a separate thread to prevent blocking the gui.

Any testing would be appreciated ;-)

Scott
